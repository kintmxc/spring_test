# 农产品溯源管理系统 - 问题解决记录

## 问题一：商品图片存储设计不合理

### 问题描述
商品表 `product` 中的 `images` 字段使用 JSON 格式存储图片 URL 数组，存在以下问题：
1. 违反数据库范式设计原则
2. 无法对单个图片进行高效查询和操作
3. 数据一致性难以维护
4. 扩展性差

### 解决方案
1. 创建独立的 `product_image` 表
2. 使用外键关联 `product` 表，支持级联删除
3. 添加排序字段 `sort_order` 支持图片排序
4. 迁移现有数据到新表

```sql
CREATE TABLE product_image (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    product_id BIGINT NOT NULL,
    image_url VARCHAR(500) NOT NULL,
    sort_order INT NOT NULL DEFAULT 0,
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_product_image_product_id 
        FOREIGN KEY (product_id) REFERENCES product(id) ON DELETE CASCADE
);
```

### 解决效果
- 数据迁移成功：4条商品图片记录
- API 兼容性保持：VO/DTO 中的 `images` 字段保留
- 查询性能提升：使用索引优化图片查询
- 数据库 `product` 表删除 `images` 列

## 问题二：订单创建失败但库存减少

### 问题描述
用户下单时出现异常情况：
- 订单创建失败
- 但商品库存已经被扣减
- 导致数据不一致

### 解决方案
将库存扣减从内存操作改为数据库乐观锁操作：

```java
// 修改前（内存操作，非原子性）
product.setStock(product.getStock() - quantity);
productMapper.updateById(product);

// 修改后（数据库乐观锁，原子性）
LambdaUpdateWrapper<Product> updateWrapper = new LambdaUpdateWrapper<>();
updateWrapper.setSql("stock = stock - " + quantity)
    .setSql("sales_count = sales_count + " + quantity)
    .eq(Product::getId, productId)
    .ge(Product::getStock, quantity);

int updatedRows = productMapper.update(null, updateWrapper);
if (updatedRows == 0) {
    throw new BusinessException("库存扣减失败，请重试");
}
```

### 解决效果
- 订单创建和库存扣减成为原子操作
- 避免了数据不一致问题
- 支持并发场景下的库存安全

## 问题三：商品图片返回 null

### 问题描述
API 返回的商品数据中 `productImage` 字段为 null，即使商品有图片。

### 解决方案
1. 创建 `ImageUtil` 工具类处理默认图片
2. 优先使用 `coverImage`，其次从 `product_image` 表查询
3. 添加完整的图片 URL 转换

```java
String imageUrl = product.getCoverImage();
if (imageUrl == null || imageUrl.isBlank()) {
    List<ProductImage> productImages = productImageMapper.selectList(
        new LambdaQueryWrapper<ProductImage>()
            .eq(ProductImage::getProductId, product.getId())
            .orderByAsc(ProductImage::getSortOrder)
    );
    if (productImages != null && !productImages.isEmpty()) {
        imageUrl = productImages.get(0).getImageUrl();
    }
}
vo.setProductImage(imageUtil.getImageUrl(imageUrl));
```

### 解决效果
- 商品图片正确返回
- 支持默认图片显示
- URL 自动补全为完整地址

## 问题五：EL1008E 缓存注解错误

### 问题描述
```
EL1008E: Property or field 'id' cannot be found on object of type 'java.lang.Long'
```

### 解决方案
错误原因：`@CacheEvict` 注解中使用了 `#result.id`，但方法返回类型是 `Long`，不是对象。

```java
// 错误写法
@CacheEvict(value = "orderDetail", key = "#result.id")
public Long create(OrderCreateDTO dto) { ... }

// 正确写法：移除该注解
public Long create(OrderCreateDTO dto) { ... }
```

### 解决效果
- 应用正常启动
- 缓存功能正常工作
- 无 EL 表达式解析错误

## 问题六：系统性能问题

### 问题描述
小程序端访问非常慢，分析原因：
- 后端问题：60%（N+1 查询、数据库连接池配置、搜索性能）
- 小程序端：15%
- 网络：25%

### 解决方案

#### 6.1 数据库连接池优化
```properties
spring.datasource.druid.initial-size=10
spring.datasource.druid.min-idle=20
spring.datasource.druid.max-active=50
spring.datasource.druid.max-wait=5000
```

#### 6.2 添加 Caffeine 缓存
```java
@Cacheable(value = "productDetail", key = "#id")
public ProductDetailVO detail(Long id) { ... }

@Cacheable(value = "orderDetail", key = "#id")
public OrderDetailVO detail(Long id) { ... }
```

#### 6.3 创建高频查询索引
```sql
-- 商品表复合索引
CREATE INDEX idx_product_sale_farmer_id ON product(sale_status, farmer_id, id);
CREATE INDEX idx_product_sale_category_id ON product(sale_status, category_id, id);

-- 订单表复合索引
CREATE INDEX idx_orders_user_status ON orders(user_id, order_status, id);
CREATE INDEX idx_orders_farmer_status ON orders(farmer_id, order_status, id);

-- 商品图片表索引
CREATE INDEX idx_product_image_sort ON product_image(product_id, sort_order);

-- 评价表索引
CREATE INDEX idx_evaluation_product_time ON evaluation(product_id, create_time, id);

-- 用户地址表索引
CREATE INDEX idx_address_user_default ON user_address(user_id, is_default, id);
```

### 解决效果
- 数据库查询效率提升
- 热点数据缓存命中率高
- 连接池资源利用合理
- 系统响应速度明显改善

## 问题七：删除订单功能缺失

### 问题描述
前端删除订单只是在本地过滤，没有真正删除数据库记录。

### 解决方案

#### 7.1 后端实现删除 API
```java
// OrderController.java
@DeleteMapping("/{id}")
public Result<Void> delete(@PathVariable Long id) {
    orderService.delete(id);
    return Result.success("删除订单成功", null);
}

// OrderServiceImpl.java
@Transactional
@CacheEvict(value = "orderDetail", key = "#id")
public void delete(Long id) {
    Orders order = getReadableOrder(id);
    if (!Integer.valueOf(OrderStatusEnum.CANCELED.getCode()).equals(order.getOrderStatus())) {
        throw new BusinessException("只能删除已取消的订单");
    }
    // 级联删除订单明细和物流信息
    orderItemMapper.delete(new LambdaQueryWrapper<OrderItem>()
            .eq(OrderItem::getOrderId, id));
    orderLogisticsMapper.delete(new LambdaQueryWrapper<OrderLogistics>()
            .eq(OrderLogistics::getOrderId, id));
    ordersMapper.deleteById(id);
}
```

#### 7.2 前端调用删除 API
```javascript
// orders.js
export const deleteOrderApi = (id) => request(http.delete(`/api/orders/${id}`))

// OrdersView.vue
async function deleteOrder(row) {
    await ElMessageBox.confirm('确认删除该订单吗？删除后无法恢复！', '删除提示')
    await deleteOrderApi(row.id)
    // 从列表中移除
    const index = records.value.findIndex((item) => item.id === row.id)
    if (index >= 0) {
        records.value.splice(index, 1)
        total.value = Math.max(total.value - 1, 0)
    }
    ElMessage.success('订单已删除')
}
```

### 解决效果
- 删除按钮只对已取消订单显示
- 删除前有确认提示
- 数据库记录真正删除
- 列表实时更新

## 问题八：schema.sql 语法兼容问题

### 问题描述
MySQL 5.7 不支持 `CREATE INDEX IF NOT EXISTS` 语法，导致应用启动失败。

### 解决方案
删除 schema.sql 中的 CREATE INDEX 语句，索引已在数据库中手动创建。

### 解决效果
- 应用正常启动
- 索引已存在于数据库中
